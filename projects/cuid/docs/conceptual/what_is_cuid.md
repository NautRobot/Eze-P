# What is CUID?

The **Component Unified Identifier (CUID or CUUID)** is generated by combining device-specific data into a 122-bit ID value. They can be either a **primary CUID** or a **derived CUID**.

## The Primary CUID
A **primary CUID** is composed as follows:

| Bits | Name | Comments |
|------|------|----------|
| 0:63 | Device Serial Number/Fingerprint | from PCIe configuration DSN caps (endpoint), SMBIOS platform serial number, ACPI or other architectural component identifier (if not available, SW generated fingerprint from proprietary individual marker) Integer, may be MSB 0-extended if smaller than 64 bit or truncated if larger |
| 64:71 | UnitID (part 1) | if component supports unitID, 0 otherwise |
| 72:79 | RevisionID | if component supports revision ID, 0 otherwise |
| 80:95 | DeviceID | from PCIe configuration space or SMBIOS ID, For CPU: Family # and Model # values combined |
| 96:111 | VendorID | from PCIe configuration space or SMBIOS vendor ID, follows PCIe vendor ID 16 bit value definition (e.g. 0x1022 or 0x1002 for AMD) |
| 112:117 | UnitID (part 2) | if UnitID value is >= 256, 0 otherwise |
| 118:121 | Device Type | See `amdcuid_device_type_t` in amd_cuid.h |

This value is then used to fill out a UUIDv8 style format as follows:

| Bit position | Value | Comments |
|--------------|-------|----------|
| 0:47 | ID value (part 1) | LSB of ID value |
| 48:51 | must be 8 (fixed) | for RFC 9562 compliance |
| 52:63 | ID value (part 2) ||
| 64:65 | 10b (fixed) | for RFC 9562 compliance |
| 66:127 | ID value (part 3) | MSB of ID value |

The serial number in particular is typically protected information, so similarly a **primary CUID** would be protected at the same level. Assuming they have the correct privilege, users should look to the **primary CUID** as the direct link between a device and the **derived CUID**.

## The Derived CUID

The **derived CUID** is what will be used for typical day to day operations. It is derived by hashing the **primary CUID** using HMAC-SHA2-256 (keyed hash) with a 256-bit key. This obscures hardware details from lower-privilege software and prevents precomputed table attacks. To protect the serial number with sensitive information, users of the tool must provide the 256-bit key to use for the keyed hashing function, as described in FIPS 198-1, The Keyed-Hash Message Authentication Code (HMAC). This key must be unique, random, and protected from unauthorized use. This scheme will result in additional management overhead to securely store the key. The key should only be accessible to privileged users or the privileged service who would then use the key along with the **primary CUID** in the HMAC-SHA2-256 function to generate the **derived CUID**. Users should note that as the key changes, for reasons such as key rotation operations, the **derived CUID** will necessarily also change. Since the **primary CUID** always remains the same however, **derived CUIDs** can always be related back to their respective device through the **primary CUID**.
